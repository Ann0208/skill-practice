## 为什么想做这个项目

做前端这些年，设计稿还原一直是个"体力活"——拿到 Figma 或 Sketch 导出的 SVG，手动拆解布局、写 CSS、调字体、对间距，反反复复。尤其是遇到复杂渐变、嵌套蒙版这类元素，光调一个色值就能耗掉半小时。更别提那些"像素级还原"的要求了，设计师拿着放大镜对比，差 1px 都得改。

去年 AI 编程工具火起来之后，我就一直在想：能不能让 AI 直接"看"设计稿，然后自动生成 HTML？不是那种生成个大概样子就完事的 Demo，而是真正能用到生产环境的、像素级还原的代码。市面上有一些设计稿转代码的工具，但大多停留在"生成个骨架"的阶段，细节还原度远远不够。

带着这个想法，我开始了 svg-to-html-toolkit 的探索。这个项目前后花了大概一个月，从最初的"让 AI 生成一版看看"，到最终搭建出一套完整的自动化转码流水线，中间踩了不少坑，也收获了很多对 AI 能力边界的认知。

## 项目是什么

一句话概括：**一个 AI 驱动的设计稿自动转码工具，能把 SVG 设计稿转成生产级 HTML/CSS，并通过"生成→验证→修正"的闭环自动迭代，直到还原度达标。**

技术栈方面：
- **后端**：Node.js + Express 5 + TypeScript，负责整个流水线的编排调度
- **浏览器渲染**：Chrome DevTools Protocol（CDP），用来把 SVG 和 HTML 分别渲染成截图做对比
- **AI 引擎**：Codex SDK，接入的是 gpt-5.5 模型，推理强度设为 high
- **OCR 文字识别**：macOS 上用 Swift Vision 原生框架（支持中英文混合），Linux 上降级为 Tesseract
- **前端**：轻量的会话管理界面，支持上传 SVG、实时查看转码进度、暂停/恢复/注入消息

整个系统的核心理念是：**不指望 AI 一次生成完美代码，而是通过自动化验证 + 迭代修正，逐步逼近目标。** 就像人类开发者写代码一样——写一版、看效果、改问题、再看效果，只不过这个循环被自动化了。

## 探索过程：从"能跑"到"能用"

### 第一步：让 AI 先生成一版

最开始的思路很简单——把 SVG 丢给大模型，让它直接输出 HTML。我用 Codex SDK 接入了 gpt-5.5 模型，给了一份 84 行的详细转换规则提示词，包括：

- 转换比例：1rem = 100px，设计稿宽度 375px（2 倍图 750px）
- 文字默认字体 PingFang SC，行高规则
- 核心原则：只用 SVG 作为数据源，不能"作弊"直接嵌入原始标记
- 渐变方向必须从 SVG 的 gradientTransform 推导，不能凭感觉写
- 复杂矢量元素（蒙版、多层渐变）保留为 SVG 资源引用

模型配置上，我把 `reasoning_effort` 设为 `high`，`sandbox` 设为 `danger-full-access`（因为需要读写文件系统），还开启了网络访问和 Web 搜索能力。每个 Agent 运行的超时时间设为 1 小时。

结果？**能跑，但惨不忍睹。** 渐变方向经常反了，文字位置偏移严重，复杂布局直接乱套。AI 生成代码的随机性比我预想的大得多——同一个设计稿跑两次，生成的 HTML 结构可能完全不同。

### 第二步：搭建三层验证流水线

光靠 AI 一次生成肯定不行，得有个"质检员"。于是我设计了一套三层验证系统，每一层检查不同维度的问题：

**第一层：像素 Diff（视觉还原度）**

这是最核心的验证。用 CDP 把原始 SVG 和生成的 HTML 分别渲染成 PNG 截图，然后逐像素对比。但这里不是简单地算"有多少像素不一样"——那样的信息对 AI 修正毫无帮助。

我做了一套精细的差异分析系统：
- **网格热力图**：把画面切成 6×12 的网格（72 个区块），统计每个区块的差异像素占比，找出差异最集中的区域
- **连通域聚类**：用 BFS 八方向连通域算法，把相邻的差异像素聚成"差异簇"（Cluster），最多追踪 12 个最大的簇。每个簇记录位置、大小、亮度趋势（HTML 比原图更亮还是更暗）、透明度趋势
- **细线检测**：专门检测低对比度的细线差异（Hairline Hotspots），这类差异肉眼几乎看不出来，但会拉高 Diff Ratio
- **水平/垂直带状分析**：把画面分成 18 个水平带和 6 个垂直带，分析差异的空间分布规律

最终生成一份 `diff-insights.md`，用自然语言描述差异情况，比如"左上角区域有一个 2400px 的差异簇，HTML 渲染比原图亮度偏高 15%，可能是渐变方向错误"。这样 AI 拿到反馈后，能精确定位问题。

**第二层：OCR 文字校验（内容正确性）**

设计稿里的文字必须一字不差。在 macOS 上我用了 Apple Vision 框架做中英文混合识别，支持按区域裁剪后识别（提高准确率）。Linux 环境降级为 Tesseract（`chi_sim+eng` 语言包）。

文字对比不是简单的字符串相等判断——OCR 识别结果经常有标点符号差异（中文逗号 vs 英文逗号）、空格差异。我用了 **bigram 相似度算法**（基于二元组的 Jaccard 相似度），先做标点和空白字符标准化，再计算相似度。差异分类为：`match`（完全匹配）、`different`（内容不同）、`html-missing`（HTML 缺少文字）、`html-extra`（HTML 多出文字）、`partial-overlap`（部分重叠）。

识别区域有两个来源：一是 HTML 中 `data-text-layout-config` 属性标记的文字区域（精确），二是从像素 Diff 的差异簇自动生成的区域（兜底）。两类区域如果重叠超过 55%，会自动合并。

**第三层：布局盒模型对比（位置精度）**

检查文字元素的位置偏移是否超过 10px。超过的标记为"几何优先级"问题，在反馈给 AI 时优先处理。这一层还会检查 HTML 的结构是否符合预期——比如是否使用了正确的容器嵌套关系。

### 第三步：闭环迭代，让 AI 自己改

验证系统搭好之后，关键的一步是**让 AI 根据验证结果自动修正**。我设计了一个动态反馈提示词生成器（`generateFeedbackPrompt`），它会：

1. 读取所有验证报告（diff-insights、text-insights、layout-box-report、workflow-lint）
2. 提取结构化的问题列表
3. 按优先级排序：**内容问题（文字不对）> 结构问题（布局错误）> 位置问题（偏移过大）> 视觉问题（颜色差异）**
4. 生成推荐的修复"配方"（Recipe），比如"建议重建容器布局，使用 cell-row 结构"
5. 附上当前 Diff Ratio 和上一轮的对比，让 AI 知道自己是在进步还是退步

每轮迭代最多自动修正 3 次（可配置），如果某一轮的 Diff Ratio 反而变大了（改进幅度不足 1%），系统会自动回滚到上一个最优版本。这个回滚机制是后来加的——因为我发现 AI 有时候"越改越差"，修了 A 问题又引入了 B 问题。

整个流水线的阶段是：`upload → preprocess → agent → font → verify → feedback → done`。每个阶段的产物都会归档到 `workflow-history/` 目录，包括当时的 HTML 代码、截图、Diff 报告、验证结果，方便回溯任意一轮的状态。

### 第四步：容器布局分析——给 AI 减负

跑了几个 Session 后我发现，Token 消耗实在太大了。每轮迭代都要把完整的 SVG 内容塞进上下文，一个稍微复杂的设计稿就是几万行 SVG 代码。

于是我加了一个**预处理阶段——容器布局分析**。在 AI 生成代码之前，先用程序分析 SVG 的树形结构：
- 识别显式分组（`<g>` 标签）
- 检测形状容器（矩形背景 + 内部元素）
- 发现重复模式（等间距排列的行/列）
- 生成对齐提示（元素间的对齐关系）

分析结果输出为一份 `container-layout.md`，用结构化的方式描述设计稿的布局层级。这样 AI 不需要从原始 SVG 的几万行代码里自己推断布局关系，直接看这份"布局说明书"就行，大幅减少了需要处理的上下文量。

### 踩过的坑

**坑一：AI 硬写 CSS 渐变。** SVG 里很多渐变效果用 CSS 是写不出来的（比如多层蒙版叠加的渐变、径向渐变配合 clipPath），但 AI 会"自信"地用 `linear-gradient` 硬写，结果完全不对。更糟糕的是，AI 有时候会花大量 Token 反复调整渐变参数，越调越离谱。

后来我加了一个**静态资源智能提取策略**——在预处理阶段自动识别复杂矢量元素（包含 `<mask>`、`<clipPath>`、多层 `<linearGradient>` 的元素），把它们提取为独立的 SVG 资源文件，在 HTML 中用 `<img>` 引用。这样 AI 只需要处理布局和简单样式，复杂视觉效果由原始 SVG 资源保证。

**坑二：Token 消耗惊人。** 12 个 Session 总共消耗了 91.4M tokens，平均每个 Session 7.6M 输入 tokens、4 万输出 tokens。输入输出比大概是 197:1——绝大部分 Token 花在了"让 AI 理解上下文"上，真正生成的代码量很少。两轮迭代的 Session 消耗大约是一轮的 2-3 倍，因为每轮都要带上之前的验证报告和修改历史。

这个成本不低。后来我通过容器布局分析减少了 SVG 上下文量，又通过更精确的反馈提示词减少了 AI 的"无效思考"，Token 消耗有所下降，但仍然是个需要持续优化的问题。

**坑三：字体渲染不一致。** CDP 渲染 HTML 时的字体效果和设计稿经常有细微差异，导致像素 Diff 虚高。同一个"PingFang SC"字体，在不同的渲染设置下，笔画粗细、字间距都会有微妙的不同。

我在启动 Chrome 时加了一系列参数来统一渲染行为：
```
--font-render-hinting=none
--disable-font-subpixel-positioning
--disable-gpu
--lang=zh-CN
```
Linux 环境还需要额外的沙箱参数。这些参数的组合是反复试出来的，每改一个参数都要重新跑一遍所有 Session 对比效果。

**坑四：会话管理的复杂性。** 一个设计稿的转码可能需要 15-30 分钟（包括多轮 AI 生成和验证），中间如果网络断了或者想暂停怎么办？我实现了完整的会话持久化机制——Session 状态实时写入磁盘（`session.json`），消息和事件用 JSONL 格式追加写入（`messages.jsonl`、`events.jsonl`）。支持暂停/恢复，恢复时从磁盘加载状态继续执行。甚至支持在运行过程中"注入消息"——比如 AI 正在生成代码时，你可以发一条消息说"注意，这个区域的背景色应该是 #FF6600"，AI 会在下一轮迭代中考虑这个信息。

前端通过 SSE（Server-Sent Events）实时接收进度更新，事件类型包括：`session:updated`（状态变更）、`step:start/complete/error`（流水线步骤）、`codex:event`（AI 生成过程的流式事件）、`pipeline:complete/error`（最终结果）。15 秒心跳保活，断线后 3 秒自动重连。

## 最终成果

经过 61 次迭代提交，成功处理了 12 个设计稿 Session：

- **最优还原度达到 97%**（Diff Ratio 0.03），这是一个相对简单的设计稿
- **平均还原度在 85-95% 之间**，复杂设计稿（多层渐变、大量文字）的还原度相对低一些
- **最差的一个 Session Diff Ratio 0.48**，是一个包含大量手写风格装饰元素的设计稿，AI 基本无法还原这类非结构化的视觉元素
- **平均每个 Session 耗时约 15 分钟**，包括预处理、AI 生成、验证、迭代修正

整个流程支持：
- WebSocket 实时进度推送 + 会话持久化 + 断点续传
- 随时暂停/恢复/注入修正指令
- 完整的工作流归档（每一轮的代码、截图、报告都保留）
- 最终产物打包下载（ZIP 格式）
- 并发控制（最多 3 个 Session 同时运行）

## 经验总结

1. **AI 生成代码不能"一锤子买卖"**，必须配合自动化验证形成闭环。单次生成的质量波动太大，但加上迭代修正后，整体效果会稳步提升。这个"生成→验证→修正"的模式，我觉得是目前 AI 编程最实用的范式。

2. **给 AI 的反馈越精确，修正效果越好。** 与其说"颜色不对"，不如说"左上角第 3 个差异簇（位置 x:120 y:45，面积 2400px）的亮度偏高 15%，HTML 渲染比原图更亮，可能是渐变方向错误"。结构化的、带空间定位的反馈，比笼统的描述有效得多。这也是为什么我花了大量时间在 Diff 分析系统上——好的反馈机制比好的生成模型更重要。

3. **知道 AI 的边界在哪里很重要。** 复杂渐变、蒙版、手写风格装饰这类视觉效果，与其让 AI 硬写 CSS，不如直接保留原始 SVG 资源。合理划分"AI 负责的部分"和"保留原始资源的部分"，整体效果反而更好。这个思路也适用于其他 AI 辅助开发场景——不要试图让 AI 做它不擅长的事。

4. **成本控制是个现实问题。** 单个设计稿转码平均消耗 7.6M 输入 tokens，12 个 Session 总共 91.4M tokens。如果按商业 API 的价格算，这个成本不低。后续可以探索的方向：用更轻量的模型做初步生成，只在精修阶段用大模型；或者通过更好的预处理减少上下文量。

5. **工程化的基础设施很重要。** 会话管理、断点续传、工作流归档、并发控制——这些"不性感"的基础设施，在实际使用中比 AI 模型本身更影响体验。一个 15 分钟的转码任务，如果中间断了要从头来，那体验就太差了。

这个项目让我对 AI 辅助前端开发有了更务实的认知——AI 不是万能的，但如果你能设计好验证机制和反馈回路，它确实能帮你把"体力活"变成"自动化流水线"。关键不在于 AI 有多聪明，而在于你给它搭建的"脚手架"有多完善。
