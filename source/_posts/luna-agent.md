基于混合搜索引擎（BM25 + 向量 + RRF 融合）和 MCP 安全 API 调用的智能客服平台，能索引后端源码，运营用自然语言提问，AI 结合代码理解给出操作指引。

<!-- more -->

## 为什么做这个

漫画后台的运营同学经常问开发各种问题："这个漫画的上架状态怎么改？""批量导入作者的入口在哪？""结算周期是怎么算的？"这些问题的答案其实都藏在后端代码里，但运营不懂代码，开发又没时间一个个解答。

我就想：能不能做一个"智能客服"，让它自动索引后端源码，运营用自然语言提问，AI 结合代码理解给出操作指引？不是那种只能回答预设 FAQ 的机器人，而是真正能"读懂代码"的助手。

## 项目核心

**一个基于混合搜索引擎 + MCP 安全 API 调用的智能客服平台**，能索引 Luna 漫画后台源码，运营人员用自然语言提问，AI 结合代码理解和实际数据查询给出操作指引。

技术栈：Node.js + TypeScript（ESM），OpenCode AI SDK 做多轮会话，BM25 + DashScope 向量嵌入做混合搜索，MCP 协议做安全 API 调用，前端是原生 HTML/CSS/JS 的暗色主题 Chat UI，部署用 Docker + PM2。

## 探索过程

### 混合搜索引擎：BM25 + 向量 + RRF 融合

运营的提问方式很口语化——"怎么改上架状态"，但代码里的描述是"UpdateComicStatus"。单靠关键词匹配（BM25）找不到，单靠语义匹配（向量检索）又可能返回不相关的结果。所以我做了混合搜索，两路并行检索再融合排序。

#### BM25 搜索与中文分词

BM25 搜索用的是 LangChain 的 BM25Retriever，但这里有个大坑——LangChain 的 BM25 内部按空格做 tokenization，英文天然有空格分隔单词，但中文文本没有空格，一整句话会被当成一个 token，完全没法匹配。比如"怎么修改上架状态"会被当成一个词，和知识库里的"上架"永远匹配不上。

我的解决方案是用 `nodejieba` 做中文分词预处理。nodejieba 是结巴分词的 Node.js 绑定，基于 C++ 实现，分词质量不错。核心思路是：在建索引时，把每个文档的内容先过一遍 jieba 分词，把"怎么修改上架状态"切成"怎么/修改/上架/状态"，再用空格拼接成"怎么 修改 上架 状态"喂给 BM25Retriever。查询时也做同样的预处理，这样 BM25 的空格分词器就能正确工作了。

分词过程中还有一个细节：jieba 会把英文字母逐个切开，比如"UpdateStatus"会被切成"U"、"pdate"、"S"、"tatus"这样的碎片。我加了一个 ASCII word buffer 机制——遇到连续的 ASCII 字符（字母和数字）就先攒在 buffer 里，遇到非 ASCII 字符或空白时再把 buffer 整体 flush 出去并转小写。这样"UpdateStatus"会被保留为完整的"updatestatus"，而不是一堆无意义的碎片。

如果 jieba 加载失败（nodejieba 依赖 C++ native 模块编译，在某些 Docker 镜像或 ARM 架构下可能编译不过），还有 `Intl.Segmenter` 作为降级方案。`Intl.Segmenter` 是 ECMAScript 标准 API，用 `new Intl.Segmenter('zh', { granularity: 'word' })` 创建实例，通过 `isWordLike` 属性过滤标点和空白。分词质量比 jieba 差一些（比如对专业术语的切分不够精准），但至少不会让服务挂掉。

#### 向量搜索与嵌入持久化

向量搜索用阿里云 DashScope 的 text-embedding-v1 模型生成嵌入向量。DashScope 提供了 OpenAI 兼容接口（baseURL 是 `https://dashscope.aliyuncs.com/compatible-mode/v1`），所以可以直接用 LangChain 的 `OpenAIEmbeddings` 类来调用，只需要把 baseURL 和 apiKey 换成 DashScope 的就行。有个限制是百炼 API 每批最多处理 10 条文本，所以 batchSize 设成了 10。

向量存储用的是 LangChain 的 `MemoryVectorStore`，纯内存实现，查询速度快。但问题是每次服务重启都要重新调 API 生成所有文档的嵌入向量——几百篇文档调一遍要好几分钟，还费 API 额度。

我做了一套持久化机制：向量索引构建完成后，把 MemoryVectorStore 内部的 `memoryVectors` 数组（包含 content、embedding、metadata）序列化成 JSON 写到磁盘。下次启动时，先算知识库所有文件的 SHA-256 哈希值，和上次保存的哈希对比——如果没变，就直接从 JSON 文件反序列化加载向量，跳过 API 调用。哈希计算会把文件名也纳入，这样新增或删除知识库文件也能被检测到。

具体来说，`computeContentHash` 函数会遍历两个知识库来源：`favorites.md`（用户收藏的 Q&A）和 `workspace/knowledge-base/*.md`（从后端源码生成的知识库文档）。所有文件内容和文件名拼接后算一个合并的 SHA-256 哈希，存在 `meta.json` 里。启动时对比这个哈希，没变就走快速加载路径，变了就全量重建索引。

#### RRF 融合排序

两路检索结果需要合并排序，我用的是 RRF（Reciprocal Rank Fusion）算法。RRF 的核心思想是：不看绝对分数，只看排名。公式是：

```
score(doc) = w_vector × 1/(k + rank_vector) + w_bm25 × 1/(k + rank_bm25)
```

其中 k=60 是 RRF 论文里的标准常数，作用是平滑排名差异——排名第 1 和第 2 的分数差距不会太大，避免某一路检索的头部结果过度主导。向量权重 0.6，BM25 权重 0.4，语义匹配优先，关键词匹配辅助。

这里有一个重要的设计决策：**RRF 只用于排序，不用于阈值判断**。判断一个结果是否"足够好"用的是原始的向量相似度分数（cosine similarity），而不是 RRF 融合分数。原因是 RRF 分数是基于排名的相对值，没有绝对含义——同样是 RRF 0.01，在一个查询里可能是很好的匹配，在另一个查询里可能完全不相关。而向量相似度是 0 到 1 之间的绝对值，0.85 就是 0.85，可以设固定阈值。

#### 三级匹配策略

融合排序后，用向量相似度分数做三级判断：

**精确匹配（score=1.0）**：查询文本和知识库里某个问题完全一致。这一步在 BM25 和向量检索之前就做了——对查询和知识库问题做规范化处理（去首尾空白、去末尾标点"？?！!。."），然后直接字符串比较。命中就直接返回缓存答案，零延迟。

**高置信匹配（向量分 ≥ 0.85）**：语义高度相似，比如用户问"漫画怎么上架"，知识库里有"如何上架漫画"，向量相似度 0.92。这种情况直接返回缓存答案，不需要额外验证。

**灰度匹配（0.65 ≤ 向量分 < 0.85）**：不太确定的区间。比如用户问"结算怎么算"，知识库里有"结算周期配置"，向量分 0.73——可能相关，也可能答非所问。这时候会调一个轻量级 LLM 做验证：把用户问题、知识库问题和答案一起发给模型，系统提示词就一句话"判断知识库的回答是否能解答用户的问题。只回答'是'或'否'，不要解释。"模型回答"是"就返回缓存答案，回答"否"就放弃缓存，交给 AI agent 自己去搜索代码回答。

低于 0.65 的结果直接丢弃，视为无关查询。

灰度区间的设计是为了平衡准确率和召回率——太严格会漏掉很多有用的匹配，太宽松又会返回不相关的结果。0.65 到 0.85 这个区间是根据实际运营提问数据调出来的。

### MCP 安全 API 调用：让 AI 查数据但不能改数据

运营问"这个漫画的结算金额是多少"时，AI 光看代码不够，还得查实际数据。但让 AI 直接调后台 API 太危险了——万一 AI 误操作删了数据怎么办？

我用 MCP（Model Context Protocol）协议搭建了一个安全的 API 调用层。MCP Server 作为子进程运行，通过 Stdio 传输层和 OpenCode SDK 通信（JSON-RPC 协议），提供两个工具：`call_api`（调用 API）和 `confirm_action`（确认操作）。

安全机制做了五层：

1. **白名单**：配置了 240+ 个只读接口（List/Get/Detail/Search 类），按服务分类——管理、结算、漫画、创作者、权限、看板等。不在白名单里的接口一律拦截
2. **危险操作检测**：DELETE 请求直接拦截，不管是否在白名单里
3. **页面权限校验**：根据用户的角色权限（从 `${PERMISSION_PAGE}` 模板变量注入）判断能不能访问这个接口
4. **参数校验**：必填字段检查，防止 AI 传了空参数导致后端报错
5. **写操作确认**：如果某个接口确实需要写操作（比如修改漫画状态），必须用户在聊天界面手动确认后才执行。`confirm_action` 工具会验证 userId 匹配，防止跨用户确认

还支持批量操作——比如运营说"帮我把张三、李四、王五都加为作者"，AI 会发一个包含三组参数的 `call_api` 调用，MCP Server 内部循环执行三次 API 调用（单次最多 20 条），返回聚合结果。

### 多轮对话查询优化器

多轮对话有个经典问题：用户第一轮问"怎么上架漫画"，第二轮问"那个按钮在哪"——"那个"指的是什么？如果直接拿"那个按钮在哪"去搜索知识库，肯定搜不到有用的结果。

我做了一个 Query Refiner（`query-refiner.ts`），在每次搜索前先用 AI 重构查询。它会看最近 6 条对话历史（为了控制上下文长度，助手回复截断到 200 字），把"那个按钮在哪"优化成"上架漫画的按钮在哪里"。

系统提示词很简洁：

```
根据对话历史和当前问题，提炼出一个独立的、适合搜索知识库的查询语句。
- 如果当前问题已足够明确，直接返回原问题
- 如果包含指代词或省略主语，结合历史补全
- 只输出提炼后的查询语句，不要解释
- 保持简洁，不超过50字
```

为了控制成本，Query Refiner 用的是一个全局共享的 session（`refinerSession`），所有用户的查询优化都复用同一个会话实例，不会为每个用户单独创建。返回结果还会清理 `<think>` 标签（有些模型会输出推理过程），并校验长度（0 < length ≤ 100 字符），不合法就回退到原始查询。

### Chat UI：暗色主题 + SSE 流式渲染

前端用原生 HTML/CSS/JS 写的（`templates/chat.html`），没有用任何框架。暗色主题设计（背景 `#1a1a2e`，强调色 `#4ecca3`），响应式布局适配不同屏幕。

消息类型有四种，渲染方式各不同：
- **用户消息**：右对齐，深蓝色背景
- **助手文本**：左对齐，带边框，支持 Markdown 渲染
- **推理过程**：斜体，绿色强调色，可折叠（默认收起）
- **工具调用**：绿色文字，显示工具名称

SSE 流式渲染的实现：前端用 `fetch` 发送 POST 请求，拿到 `ReadableStream` 后逐 chunk 解析。文本内容按 20 字符一组追加渲染（避免逐字符渲染导致的频繁 DOM 操作）。推理内容和正文内容分开处理——检测到 `<think>` 标签就切换到推理模式，检测到 `</think>` 就切回正文模式。

每条消息下面有点赞/点踩按钮，反馈数据发送到 `/helper/favorite` 或 `/helper/dislike` 接口，用于后续的质量监控和知识库优化。还有三个导航链接：搜索分析看板（`/helper/stats`）、反馈审核（`/helper/review`）、差评列表（`/helper/dislikes`）。

### Docker + PM2 部署

项目通过 Docker 容器化部署。基础镜像是 `hub.bilibili.co/nyx-base/nodejs-debian12:base`，构建步骤：

1. 安装系统依赖：curl、git、ca-certificates、ripgrep、nodejs、npm
2. 安装全局工具：pnpm@10.11.0、pm2、opencode-ai
3. 复制 package 文件 → `pnpm install --frozen-lockfile`
4. 复制源码 → `pnpm build`（TypeScript 编译）
5. 用 git token 克隆 Luna 后台源码（作为知识库）
6. 创建 knowledge-base 目录
7. 暴露端口 3400

PM2 负责进程管理，配置了自动重启（最多 30 次）和 2 秒重启延迟。开发环境用 `tsx` 直接运行 TypeScript，生产环境用编译后的 `dist/index.js`。

有个环境兼容性问题排查了一阵子：Docker 容器的内核版本是 5.10，不支持 Landlock 安全沙箱（Linux 5.13+ 才支持），OpenCode SDK 启动时会报错。解决方案是设置环境变量 `OPENCODE_DISABLE_SANDBOX=1`。

MCP 连接也有冷启动问题——Docker 容器刚启动时 OpenCode Server 还没准备好，MCP 连接会失败。我设了 30 秒的首次等待，加上 3 次后台重试（每次间隔 10 秒）。大部分情况下第一次重试就能连上。

### 搜索日志与分析

每次搜索都会记录详细的日志（`search-logger.ts`），JSONL 格式追加写入：

```json
{
  "timestamp": "2026-04-15T10:30:00Z",
  "query": "怎么上架漫画",
  "refinedQuery": "如何上架漫画",
  "matchType": "confident",
  "totalDurationMs": 234
}
```

`matchType` 有六种：`exact`（精确匹配）、`confident`（高置信）、`verified`（灰度验证通过）、`rejected`（灰度验证未通过）、`no_match`（无匹配）、`agent`（AI 自主回答）。通过分析这些日志，可以知道知识库的覆盖率、哪些问题经常被问到但没有缓存答案、灰度验证的准确率等。

搜索分析看板（`/helper/stats`）会可视化展示这些数据，帮助持续优化知识库。

## 项目成果

80+ 次迭代提交，交付了完整的智能客服平台：

- **智能问答**：混合搜索（BM25 + 向量 + RRF）+ AI 代码理解，支持多轮对话
- **三级匹配策略**：精确匹配零延迟、高置信直接返回、灰度区间 LLM 验证
- **安全 API 调用**：240+ 白名单接口，五层安全机制，写操作需人工确认
- **查询优化器**：自动消除多轮对话中的指代歧义
- **搜索分析看板**：命中率、匹配类型分布、热门问题统计
- **反馈审核系统**：用户点赞/点踩，管理员审核反馈质量
- 已通过 Docker + PM2 部署上线，Luna 漫画运营团队日常使用

## 经验总结

1. **混合搜索是 RAG 场景的标配。** BM25 抓关键词，向量抓语义，RRF 融合后效果远好于单一方式。但要注意：RRF 只管排序，阈值判断要用原始的向量相似度分数，不能混用。

2. **中文 NLP 的基础设施比英文差很多。** LangChain 的 BM25 默认按空格分词，对中文完全不友好。类似的坑在很多英文优先的 AI 框架里都有，用之前一定要验证中文场景的表现。

3. **AI 调用外部 API 必须有安全边界。** 白名单 + 权限校验 + 写操作确认，缺一不可。"AI 不会犯错"是最危险的假设。MCP 协议提供了一个很好的安全隔离层——AI 只能通过预定义的工具接口访问外部系统，不能直接发 HTTP 请求。

4. **多轮对话的查询优化容易被忽视。** 用户的第二轮、第三轮提问往往有指代和省略，不做查询重构的话，搜索质量会断崖式下降。一个简单的 Query Refiner 就能大幅改善体验。

5. **知识库的变更检测很实用。** 用文件哈希判断知识库是否更新，没变就跳过重建索引，既省 API 调用费用，又加快启动速度。这个模式在任何需要"索引外部数据"的场景都适用。

6. **灰度匹配策略是个好思路。** 不是所有查询都能明确判断"匹配"或"不匹配"，中间地带用轻量级 LLM 做二次验证，比硬设阈值灵活得多。成本也可控——只有落在灰度区间的查询才会触发 LLM 调用。
