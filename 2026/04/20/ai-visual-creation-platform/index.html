2400+ 次提交的 AI 可视化创作平台：Canvas 编辑器与 Agent 对话实践

发表于2026-04-20|更新于2026-04-29|AI 项目实战
|浏览量:

团队协作的大型 AI 可视化创作平台，我负责 Canvas 节点编辑器、AI 对话面板、SSE 流式通信、积分系统前端。2400+ 次提交，微服务架构。

背景

前面五个项目都是我独立从 0 到 1 做的，这个不一样——它是一个团队协作的大型项目，我作为前端核心开发参与。是一个 AI 可视化创作平台，用户可以在画布上通过节点连线编排 AI 生成流程（图片、视频、文本），Agent 通过 Tool Calling 自动执行生成任务并把结果回填到画布节点。

之所以想聊这个项目，是因为它的规模和复杂度跟个人项目完全不是一个量级——2400+ 次提交、微服务架构、多 AI Provider 集成、积分计费系统、桌面客户端。在这样的项目里做前端，遇到的问题和个人项目很不一样。

项目全貌：微服务架构

整个平台是微服务架构，四个服务各司其职：

  • 前端（Vite Dev :5173 / Nginx）：React 19 + Vite 6 + Zustand + Ant Design 6 + TailwindCSS，承载画布编辑器、对话面板、素材管理、Skill 市场等所有用户界面
  • 后端（FastAPI :8000）：项目管理、素材管理、用户认证、积分系统、异步生成任务调度，是整个平台的数据中枢
  • Agent 服务（FastAPI :8001）：AI 对话、ReAct 循环、Tool Calling、SSE 流式通信，独立部署是因为 Agent 任务耗时长且资源消耗大，需要和主后端隔离
  • 桌面客户端（PySide6 + QML）：本地生成，不消耗云端额度，面向有本地 GPU 或自有 API Key 的用户

服务间的通信设计是我觉得比较清晰的一点。前端请求通过 Nginx 路径分发——/api/* 代理到后端 :8000，/agent/* 代理到 Agent 服务 :8001，前端代码里不需要关心实际端口。Agent 服务需要调用后端接口（比如保存生成结果、扣减积分），走的是内部认证——请求头带 X-Internal-Key，后端校验这个 Key 来区分”用户请求”和”服务间调用”，权限模型不同。桌面客户端和后端之间通过 SSE 长连接通信，后端推送任务状态变更，客户端接收后执行本地生成并回传结果。

这种架构的好处是职责清晰、可以独立扩缩容。Agent 服务在高峰期可以单独扩容，不影响主后端的稳定性。坏处是联调成本高——前端一个功能可能同时依赖两个后端服务，本地开发要同时起三个进程。

我负责的核心模块

Canvas 节点编辑器

这是整个平台的核心交互——用户在画布上拖拽节点，用连线表示数据流转关系，比如”文本节点生成提示词 → 图片节点生成图片 → 视频节点生成视频”。听起来简单，但做下来发现画布编辑器是前端复杂度最高的模块，没有之一。

状态管理：1400 行的 Canvas Store

用 Zustand 写的 canvasStore.ts，大概 1400 行，统一管理节点位置、连线关系、选区状态、历史栈（撤销/重做）、生成任务状态、素材列表。这个 Store 是整个前端最复杂的部分——每次拖拽、缩放、连线都要更新状态，还要保证性能不卡。

为什么不拆成多个小 Store？试过，但画布操作经常需要同时更新多个状态切片（比如拖拽节点时要更新节点位置、连线端点、选区范围），拆开后跨 Store 协调反而更复杂。集中管理虽然文件大，但数据流清晰。

拖拽性能：DOM 优先，状态延迟

画布交互的性能瓶颈在于：拖拽过程中每一帧都要更新节点位置，如果每帧都触发 Zustand 状态更新 → React 重渲染，帧率会掉到个位数。

我的做法是拖拽过程中只更新 DOM，不更新 Store。用 @dnd-kit 处理拖拽交互，拖拽过程中直接操作 DOM 的 transform 属性（零状态更新），拖拽结束后才一次性提交到 Zustand Store。缩放和平移也是类似的策略——100ms 防抖后才同步状态。这样拖拽过程中完全不触发 React 重渲染，体验丝滑。

四种节点类型

节点类型有四种，每种有不同的交互逻辑和渲染方式：

  • 图片节点：支持生成参数配置（模型选择、提示词、参考图）和结果预览，生成完成后自动更新封面
  • 视频节点：支持文生视频（t2v）和图生视频（i2v）两种模式，生成过程中显示进度百分比
  • 文本节点：用 Tiptap 做富文本编辑，支持 Markdown 快捷输入
  • Agent 节点：嵌入一个完整的对话界面，用户可以在画布上直接和 AI 对话

AI 对话面板和 SSE 流式通信

对话面板是用户和 AI Agent 交互的入口。我封装了一个 useSSE Hook（hooks/useSSE.ts）来处理流式通信——没有用浏览器原生的 EventSource（因为它不支持 POST 请求和自定义 Header），而是用 fetch + ReadableStream 手动解析 SSE 流。

SSE 事件类型有 7 种以上：

  • thinking_reset：AI 开始新一轮思考，清空之前的思考内容
  • progress：工具执行进度（比如”正在生成图片…”）
  • text_delta：文本增量，追加到对话气泡
  • result：最终结果（包含完整的回复内容）
  • image_saved：图片生成完成，触发画布节点刷新
  • task_complete：整个任务结束
  • error：错误信息

流式通信最头疼的是断线恢复。我做了三层保障：

  1. SSE 实时推送（主路径）：正常情况下通过 SSE 接收所有事件
  2. 4 秒轮询降级（备用）：SSE 连接断开时自动切换到轮询模式，每 4 秒查询一次任务状态
  3. 页面刷新恢复（兜底）：从后端重新加载完整的对话历史

这三层保障的背后是 Agent 服务的 StreamHolder 架构——它把 Agent 任务和 SSE 连接解耦了。任务在后台持续运行（由 TaskManager 管理），SSE 连接断了可以通过 GET /agent/v2/conversations/{id}/subscribe 重新订阅，不会丢失中间的事件。这对长时间的图片/视频生成任务特别重要——一张图可能要生成 30 秒，中间网络抖动一下不能让整个任务白费。

对话历史用 processedHistoryRef 维护完整记录，但渲染时只展示最近 20 条消息（虚拟滚动），避免长会话导致 DOM 节点过多影响性能。

还有一个上下文压缩机制：当对话历史的 Token 使用率达到 80% 时，自动触发压缩——用 LLM 生成一份对话摘要，替换掉旧的消息。压缩后的摘要存在数据库的 context_summary 字段里，下次加载对话时直接用摘要恢复上下文。

积分三阶段计费系统

平台的 AI 生成功能是收费的，用积分计费。积分系统的核心是三阶段事务：

  1. 冻结（Freeze）：用户发起生成请求时，根据模型和参数估算费用，从余额中预扣（balance -= amount, frozen += amount）。如果余额不足，直接拒绝请求
  2. 结算（Settle）：生成完成后，按实际用量计算费用（可能比预估多或少），从冻结金额中扣除实际费用
  3. 释放（Release）：生成失败时，把冻结的金额退还到余额（frozen -= amount, balance += amount）

UserCredit 表用了乐观锁（version 字段）防止并发更新冲突——每次更新余额时检查 version 是否变化，变了就重试（最多 3 次）。

前端需要实时展示”可用余额”和”冻结金额”，但这两个值在生成任务进行中会频繁变化。我用乐观更新的策略——发起生成请求时立即在前端扣减余额显示，任务完成后再用后端返回的实际值校正。这样用户不会看到余额”先不变、突然跳变”的奇怪体验。

后端还有一个 TaskPool（基于 Redis Lua 脚本实现原子操作），按用户维度限制并发：图片最多 5 个、视频最多 2 个、文本最多 3 个同时生成。超出限制的任务进入队列（每用户每类型最多排队 20 个），有空位时自动出队执行。

Skill 市场

平台支持用户上传自定义 Skill（技能），扩展 Agent 的能力。Skill 本质上是一个包含 SKILL.md（frontmatter + 指令内容）的目录，打包成 ZIP 上传。

上传后的审核流程：

  1. AST 安全扫描：解析 Skill 代码，检查是否导入了危险模块（os、sys、subprocess 等被禁止的 import）
  2. AI 审核（可选）：用 LLM 评估 Skill 的安全性
  3. 人工审核：管理员在后台审批/拒绝
  4. 发布：设置 is_private=False，其他用户可以在市场浏览和激活

用户激活 Skill 后，Agent 在下次对话时会自动加载该 Skill 的指令和工具定义。用户 Skill 的优先级高于内置 Skill（overwrite=True），可以覆盖默认行为。

桌面客户端：本地生成不花钱

桌面客户端用 PySide6（Qt for Python）+ QML 开发，面向有本地 GPU 或自有 API Key 的用户。核心功能是：用户配置本地的 API Key（存在系统 Keychain 里，macOS 用 Keychain，Windows 用 Credential Manager），然后 Agent 会自动检测桌面客户端是否在线（通过心跳机制，120 秒超时），优先使用本地配置生成图片/视频，不消耗服务端积分。

这个设计挺巧妙的——对平台来说减少了服务端算力消耗，对用户来说省了积分。Agent 服务在执行 image_generate_async 工具时，会先查 Redis 里用户的桌面客户端配置（dc:user:configs:{user_id}），如果 agent_use_local=true 且客户端在线，就把任务分发到桌面客户端执行。

踩过的坑

Canvas 性能优化。 一开始节点多了之后（50+ 个）画布明显卡顿，原因是每次状态更新都触发所有节点重渲染。用 React.memo 包裹节点组件 + 细粒度的 Zustand selector 优化——每个节点只订阅自己相关的状态切片（store.nodes[nodeId]），其他节点的变化不会触发重渲染。优化后 100+ 节点也能流畅操作。

SSE 和 Axios 的冲突。 项目用 Axios 做 HTTP 客户端，但 Axios 不支持流式响应（它会等整个响应完成才返回）。SSE 请求只能绕过 Axios，直接用 fetch。这导致了两套请求方式并存——普通 API 用 Axios（自动注入 JWT、401 自动刷新 Token），SSE 用 fetch（需要手动处理认证）。认证 token 的注入逻辑要维护两份，容易出 bug。

积分系统的前端状态同步。 三阶段计费意味着余额在”冻结→结算/释放”过程中会多次变化。如果前端每次都等后端返回再更新，用户会看到余额”跳来跳去”。乐观更新解决了大部分问题，但还有一个边界情况：用户同时发起多个生成任务，每个任务的冻结/结算/释放事件交错到达，前端的乐观值和后端的实际值可能出现偏差。最终的解决方案是：乐观更新只用于”冻结”阶段（立即扣减显示），”结算”和”释放”阶段直接用后端返回的实际值覆盖。

项目成果

参与 2400+ 次迭代提交，交付的核心模块包括：

  • Canvas 节点编辑器：四种节点类型，拖拽/连线/缩放/撤销重做，DOM 优先渲染策略
  • AI 对话面板：SSE 流式通信，三层断线恢复，虚拟滚动
  • 多模态生成集成：图片（Bilibili LLM / DashScope / Ark）、视频（PPIO / Kling / Veo）、文本
  • Skill 市场前端：社区浏览、上传管理、激活/评分/评论
  • 积分系统前端：余额展示、充值、消费记录、乐观更新

整个平台还包括桌面客户端（PySide6），支持本地生成不消耗云端额度。多 AI Provider 集成（Bilibili LLM、DashScope、PPIO、Ark、Kling），覆盖图片/视频/文本三种生成类型。

经验总结

  1. 大型项目的状态管理，”少即是多”。 Canvas Store 1400 行听起来很多，但它把所有画布相关的状态集中管理，比分散在各个组件里清晰得多。关键是用 selector 做细粒度订阅，避免不必要的重渲染。Zustand 的 useStore(store, selector) 模式在这种场景下比 Redux 轻量很多。

  2. SSE 流式通信在生产环境比想象中复杂。 断线重连、任务恢复、事件去重、连接和任务解耦……每一个都是实际踩过的坑。StreamHolder 架构（任务和连接分离）是解决长时间任务断线问题的关键设计。

  3. 前后端协同设计协议很重要。 SSE 事件格式、错误码定义、状态流转——这些如果前端不参与设计，后端定义出来的协议往往不好消费。前端最清楚”UI 需要什么数据、什么时机”，这些信息应该在协议设计阶段就输入进去。

  4. 乐观更新要谨慎。 简单场景下乐观更新体验很好，但涉及多步骤事务（冻结→结算→释放）和并发操作时，乐观值和实际值的偏差会让用户困惑。需要明确哪些阶段用乐观更新、哪些阶段用服务端真实值覆盖。

  5. 参与大型项目和独立做小项目，成长维度不同。 小项目锻炼的是从 0 到 1 的全栈能力——什么都要自己做，快速出活。大项目锻炼的是协作、架构理解、性能优化——在已有的架构约束下做出好的设计决策，和后端同学对齐协议，在不影响其他模块的前提下优化自己负责的部分。两种经历互补，缺一不可。
