LangChain + FAISS 实战：给需求文档做一次 AI "体检"

发表于2026-04-01|更新于2026-04-29|AI 项目实战
|浏览量:

基于 LangChain/LangGraph 的需求质量分析 WebUI 工具，支持 0-5 分质量评分、FAISS 向量检索关联历史需求、对话式 Diff 编辑。Monorepo 架构，38 次迭代提交。

起因

做开发的应该都经历过这种场景：产品经理写了个需求，标题是”优化搜索功能”，描述就一句话”搜索结果不够准确，需要优化”。然后你问：优化哪个页面的搜索？什么叫”不够准确”？有没有具体的验收标准？一来一回沟通半天，需求评审会变成需求补充会。

更头疼的是，有时候两个需求之间存在隐含冲突——A 需求说搜索按时间排序，B 需求说按相关度排序——但因为是不同的产品经理写的、隔了几个月，没人记得。等到开发阶段才发现，返工成本就上去了。

我就想：能不能在需求提交之前，先让 AI 帮忙”体检”一下——哪些信息缺失、质量打几分、有没有跟历史需求冲突？这样产品经理自己就能查漏补缺，不用等到评审会上被开发追问。

这就是 tapd-ai-helper 的出发点。

项目核心

一个基于 LangChain/LangGraph 的需求质量分析 WebUI 工具，核心功能包括：需求质量评分（0-5 分）、开发摘要提取、历史需求多模态关联分析、对话式需求编辑（Diff 预览 + 一键应用）。

技术栈：前端 React 17 + TinyMCE 富文本编辑器 + Vite 6，后端 Node.js + Express 5 + TypeScript（ESM），AI 层用 LangChain + LangGraph 做 Agent 编排，FAISS 做向量检索。整个项目用 pnpm workspace 搭建 Monorepo，前后端独立开发和部署。

Monorepo 工程结构

项目用 pnpm workspace 管理，根目录的 pnpm-workspace.yaml 声明了两个包：apps/web（前端）和 apps/server（后端）。这样做的好处是前后端可以共享 TypeScript 类型定义——比如 API 的请求/响应类型、需求数据结构、评分结果结构，定义一次两边都能用，不会出现前后端字段名不一致的低级问题。

apps/web 是 Vite 6 + React 17 的 SPA，核心页面就一个：左边是 TinyMCE 富文本编辑器（展示和编辑需求内容），右边是 ChatTab 聊天面板（和 AI 交互）。apps/server 是 Express 5 + TypeScript ESM 的后端，负责调用 LangGraph 工作流、管理 FAISS 向量库、处理 Diff 计算。

开发时 pnpm dev 一条命令同时启动前后端的 dev server，前端通过 Vite 的 proxy 配置把 /api 请求转发到后端，体验很流畅。部署时各自独立构建，前端产出静态文件，后端产出 Node.js 服务。

探索过程

LangGraph 多节点工作流

这是我第一次用 LangGraph，它的核心概念是”图”——把 AI 任务拆成多个节点，节点之间通过共享状态传递数据。

先说状态管理。LangGraph 用 Annotation.Root() 定义整个图的共享状态类型。你可以把它理解成一个 TypeScript interface，但它不只是类型声明——每个字段还可以指定 reducer 函数，决定当多个节点写同一个字段时怎么合并。比如我的综合分析图里，qualityResult 和 conflictResult 是两个独立节点分别写入的，通过 Annotation.Root() 定义后，LangGraph 会自动管理状态的读写和传递，不需要手动在节点之间传参。

一开始我没理解 Annotation.Root() 的工作方式，直接在节点函数里 return 了一个新对象，结果前一个节点写入的状态被覆盖了。后来才搞明白：节点返回的是”增量更新”，LangGraph 会把返回值 merge 到现有状态上，而不是替换整个状态。这个心智模型和 React 的 setState 很像。

我设计了四个独立的工作流图：

需求摘要图：一个节点，输入需求名称和描述，输出精炼标题、核心摘要（100 字以内）、关键变更点、影响范围、工程风险点。这个给开发看，一眼就知道这个需求要改哪里、有什么风险。

质量评审图：一个节点，按 0-5 分打分。评分标准我花了不少时间设计（下面单独展开讲）。

关联分析图：一个节点，结合历史需求库，找出与当前需求相关的历史需求，按关联类型分类——同模块、上下游依赖、规则继承、潜在重叠、数据耦合。每条关联都要求 AI 引用具体的文本证据，不能空口说”可能相关”。

综合分析图：两个节点串联，先做质量评审，再做冲突检测。冲突检测会找出当前需求和历史需求之间矛盾的地方，比如”A 需求说搜索结果按时间排序，B 需求说按相关度排序”。这里体现了 LangGraph 的优势——两个节点通过 Annotation.Root() 共享状态，质量评审节点的输出自动成为冲突检测节点的输入，不需要手动编排数据流。

LangGraph 的好处是每个节点独立处理和错误捕获，一个节点挂了不影响其他节点。而且图的结构天然支持并行——如果两个节点之间没有依赖关系，LangGraph 会自动并行执行。

质量评分体系：门槛项 vs 加分项

评分标准是这个项目里我打磨最久的部分。最初的版本很简单——列了十几个检查项，每项 0.5 分，加起来就是总分。但很快发现问题：一个需求如果核心意图都说不清楚，但恰好写了异常处理和验收标准，也能拿到 3 分以上，这明显不合理。

所以我把评分标准分成了两层：门槛项和加分项。

门槛项是”没有就不及格”的硬性要求：

  • 能看懂要做什么：需求的核心意图是否清晰？读完之后能不能用一句话概括”这个需求要实现什么”？
  • 能定位改哪里：涉及哪个模块、哪个页面、哪个接口？开发拿到后能不能知道从哪里开始改代码？
  • 核心业务规则清晰：主流程的逻辑是否完整？比如”满 100 减 20”，触发条件、计算规则、生效范围是否都写了？

加分项是”有了更好”的软性要求：

  • 原型图或 UI 设计稿
  • 交互细节（loading 状态、空状态、异常提示）
  • 明确的验收标准
  • 异常场景和边界条件
  • 数据埋点需求
  • 性能要求

具体到每个分数段，我给 AI 的 prompt 里定义了锚点：

  • 5 分：开发拿到就能直接拆任务排期。核心流程、异常场景、验收标准、交互细节全覆盖，不需要额外沟通。
  • 4 分：核心流程完整清晰，有验收标准，但可能缺少一些边界场景或异常处理的细节，开发需要少量确认。
  • 3 分：核心意图明确，能看懂要做什么，但有 1-2 处关键信息需要补充——比如缺少具体的业务规则、或者影响范围不明确。
  • 2 分：能大致猜到方向，但关键信息缺失严重，开发需要大量沟通才能开始。
  • 1 分：只有一个模糊的想法，几乎没有可执行的信息。
  • 0 分：完全看不懂要做什么，或者内容为空。

这套分层评分的好处是：门槛项不达标，分数天花板就被压低了，不会出现”细节丰富但核心不清”的高分需求；同时加分项的缺失不会过度惩罚，避免了”因为没写埋点需求就扣分”的不合理情况。

FAISS 向量检索 + 多模态混合检索

历史需求关联分析需要从上千条历史需求中找出相关的。单靠关键词匹配太粗糙，我用了三路并行检索再融合的策略。

第一路：向量语义检索。 用 FAISS 做语义相似度匹配。把每条需求的”名称 + 描述”拼接后生成向量（用的 Ollama 本地跑的 bge-m3 模型，768 维），存到 .vector-store/ 目录下的 FAISS 索引文件。查询时把当前需求文本同样编码成向量，用余弦相似度检索 Top-8 结果。FAISS 的优势是纯本地运行、速度快，上千条需求的检索在毫秒级完成。

第二路：词法检索。 对需求文本做中文分词（bigram/trigram），计算词汇重叠度。这里有个权重设计：名称匹配权重 ×4，描述匹配权重 ×1。为什么名称权重这么高？因为需求名称通常是高度浓缩的关键信息，两个需求如果名称里有相同的关键词（比如都提到”搜索结果排序”），关联概率很高。这一路能捕捉到向量检索可能遗漏的精确关键词匹配——向量检索擅长”意思相近”，但有时候两个需求用了完全不同的表述方式描述同一个功能，向量距离反而不近。

第三路：多模态关键词匹配。 这是比较有意思的一路。很多需求文档里贴了截图（原型图、UI 设计），纯文本检索完全忽略了图片信息。我预先对历史需求的图片做了语义分析——用视觉模型提取图片描述、OCR 文字、UI 元素、交互行为、约束条件，存成结构化的 ImageSemanticInfo。检索时，文本关键词匹配权重 ×2，图片关键词匹配权重 ×5。为什么图片权重这么高？因为图片信息更稀缺，如果当前需求的关键词在某条历史需求的截图里出现了，说明关联度很高——比如当前需求提到”订单列表页”，而某条历史需求的原型图里正好有订单列表的截图。

三路融合策略： 三路检索各自返回带分数的候选列表，合并时按需求名称去重——如果同一条需求被多路检索命中，保留最高分。最终取 Top-10 作为候选集，交给 AI 做精细的关联分析。AI 会对每条候选需求判断关联类型（同模块、上下游依赖、规则继承、潜在重叠、数据耦合），并引用具体的文本证据。

对话式需求编辑：Diff 预览 + 一键应用

质量评审发现问题后，下一步自然是”改”。我做了一个对话式编辑功能——用户在聊天框里说”把验收标准补充完整”，AI 会生成修改建议，以 Diff 的形式展示（哪些内容新增、哪些删除、哪些修改），用户确认后一键应用到富文本编辑器。

Diff 生成：LCS 算法 + HTML 预处理。 Diff 的核心是 LCS（最长公共子序列）算法，按行对比原文和修改后的文本，找出哪些行是共同的、哪些是新增的、哪些是删除的。但需求文档是 HTML 格式，直接按行切分会把 <p> 标签切碎，所以我先做了一步 HTML 标准化预处理——在块级标签（</p>、</div>、</li> 等）后面插入换行符，确保每个语义块独占一行，再做行级对比。行相似度阈值设为 85%（用 Levenshtein 距离计算），低于这个值就判定为”不同行”而不是”修改行”，避免把两段完全不同的内容误判为”修改”。

变更分组：splitIntoChangeGroups。 AI 一次可能给出多处修改建议，比如同时补充了验收标准、修改了业务规则、增加了异常场景。我把 Diff 结果中连续的插入/删除操作分组——相隔不超过 5 行的变更合并为一组，每组代表一个独立的修改点。这样用户可以按组查看，每组都有清晰的上下文，不会看到一堆零散的红绿标记。

部分应用：applyPartialDiff。 用户可以选择只应用其中几组修改，而不是全部接受。这个实现比想象中复杂——因为应用前面的修改会改变行号偏移，后面的修改需要重新计算位置。我的做法是从后往前应用（倒序），这样前面的行号不会被影响。

安全机制：base content 校验。 这是一个容易忽略但很重要的点。AI 生成 Diff 时，是基于当时的需求内容计算的。但用户可能在 AI 生成建议后又手动改了内容——比如自己先改了一段，然后再点”应用 AI 建议”。这时候 Diff 的基准已经变了，强行应用可能会覆盖用户的手动修改，甚至产生乱码。所以应用 Diff 前，后端会校验当前编辑器里的内容是否和生成 Diff 时的 base content 一致，不一致就报错提示用户重新生成建议。

前端实现：React + TinyMCE + 流式聊天

前端的核心交互是左右分栏：左边 TinyMCE 富文本编辑器展示需求内容，右边 ChatTab 聊天面板和 AI 交互。

ChatTab 支持流式输出（SSE），AI 的回复是逐字显示的，体验比等半天再一次性显示好很多。流式处理的关键是正确处理 SSE 的 chunk 拼接——后端用 text/event-stream 格式推送，前端用 EventSource 或 fetch + ReadableStream 接收，每收到一个 chunk 就追加到当前消息的内容里，触发 React 重新渲染。

当 AI 返回的是修改建议时，ChatTab 会渲染一个 proposal 预览组件，展示 before/after 的 HTML 对比。新增的内容用绿色背景高亮，删除的用红色背景加删除线，修改的用黄色背景。每组变更旁边有”应用”和”忽略”按钮，用户可以逐组决定。点击”应用”后，对应的修改会实时反映到左边的 TinyMCE 编辑器里。

TinyMCE 的集成有个细节：它的内容更新需要通过 editor.setContent() API，而不是直接操作 DOM。而且 setContent() 会触发 onChange 事件，如果不做防抖处理，会导致”应用 Diff → 触发 onChange → 重新计算 Diff”的死循环。我用了一个 isApplyingDiff 的 flag 来屏蔽应用期间的 onChange 回调。

踩过的坑

结构化输出的兼容性。 LangChain 的 withStructuredOutput 在不同模型上表现差异很大，这是我踩得最深的一个坑。理想情况下，你定义一个 Zod schema，调用 model.withStructuredOutput(schema)，模型就会返回符合 schema 的 JSON 对象。但现实是：有的模型（比如 GPT-4）支持 function calling 模式，能严格按 JSON Schema 约束输出；有的模型只支持 response_format: { type: "json_object" }，能保证返回合法 JSON 但不保证字段结构；有的模型（特别是一些开源模型通过 Ollama 跑的）干脆不支持任何结构化输出约束，返回的是 Markdown 格式的文本，JSON 藏在代码块里。

我做了三层降级策略：第一层，尝试 withStructuredOutput（JSON Schema 模式）；如果报错或返回 null，降级到第二层，用 response_format: json_object 模式调用，拿到 JSON 后手动校验字段；如果还是失败，降级到第三层，让模型自由输出，然后用正则从 Markdown 的 ```json 代码块里提取 JSON 字符串，再 JSON.parse。三层都失败才报错。实测下来，这套降级策略覆盖了我测试过的所有模型（GPT-4、GPT-3.5、Claude、Qwen、DeepSeek、Llama 系列）。

TinyMCE 和 Diff 的冲突。 TinyMCE 会自动”美化” HTML——比如把 <br> 变成 <br />，把 <img src=x> 变成 <img src="x" />，把多余的空行合并，甚至会重新排列标签属性的顺序。这导致 Diff 对比时出现大量无意义的差异——明明内容没变，但 HTML 源码变了。我在对比前做了 HTML 标准化处理：统一自闭合标签格式、移除多余空白、标准化属性引号。标准化之后再做 Diff，干净多了。

FAISS 索引的持久化和更新。 FAISS 索引是内存数据结构，需要手动序列化到磁盘。一开始我每次启动服务都重新构建索引，上千条需求要跑几分钟。后来改成增量更新——启动时加载磁盘上的索引文件，只对新增的需求生成向量并追加到索引里。但 FAISS 的 IndexFlatIP（内积索引）不支持删除操作，如果历史需求被修改了，旧的向量还在索引里。我的处理方式是定期全量重建（比如每天凌晨），日常使用增量追加，容忍少量过期数据。

SSE 流式输出的错误处理。 流式输出时如果 AI 调用中途报错（比如 token 超限、网络中断），前端已经渲染了一半的内容。我的处理是：后端在 SSE 流里发送一个特殊的 error event，前端收到后在当前消息末尾追加一个错误提示，而不是清空已有内容。这样用户至少能看到 AI 已经生成的部分内容，不至于什么都没有。

项目成果

38 次迭代提交，交付了四大核心功能：

  • 需求质量评分：0-5 分量化评估，门槛项 + 加分项分层评价，每个分数段有明确的锚点定义
  • 开发摘要提取：一键生成精炼标题、关键变更、影响范围、工程风险，开发拿到就能评估工作量
  • 历史需求关联分析：三路混合检索（向量 + 词法 + 多模态），支持图片语义匹配，按名称去重取最高分
  • 对话式编辑：AI 生成修改建议，Diff 预览支持逐组查看，部分应用，base content 安全校验

Monorepo 架构让前后端可以独立开发和部署，共享类型定义，pnpm workspace 管理依赖干净利落。

经验总结

  1. LangGraph 适合”有明确步骤”的 AI 任务。 如果你的任务可以拆成几个独立的分析步骤，用图的方式编排比写一个巨大的 prompt 清晰得多。每个节点职责单一，调试也方便——某个节点输出不对，直接看那个节点的输入输出就行，不用在一个几千 token 的 prompt 里找问题。Annotation.Root() 的状态管理机制也很优雅，理解了”增量更新”的心智模型之后用起来很顺手。

  2. 多路检索融合比单一检索好得多。 向量检索擅长语义匹配但可能漏掉精确关键词，词法检索擅长精确匹配但不懂语义，图片信息更是纯文本检索的盲区。三路融合后，召回质量明显提升。关键是融合策略要简单——按名称去重、保留最高分，不要搞复杂的加权公式，简单的策略反而鲁棒性更好。

  3. AI 生成的结构化输出一定要做降级处理。 不同模型、不同版本的结构化输出能力差异很大，不能假设”模型一定会返回合法 JSON”。三层降级（JSON Schema → json_object → 手动提取）是必须的，而且要在每一层都做好日志记录，方便排查到底降级到了哪一层。

  4. Diff 应用到富文本比纯文本复杂得多。 HTML 的标签嵌套、自动格式化、空白字符差异，都会干扰 Diff 算法。预处理和标准化是关键。另外，部分应用 Diff 时要注意行号偏移的问题，倒序应用是最简单的解决方案。

  5. 流式输出的细节决定体验。 SSE 本身不难，但错误处理、中断恢复、防抖渲染这些细节加起来，工作量不小。建议一开始就把错误处理的框架搭好，不要等到最后再补。
