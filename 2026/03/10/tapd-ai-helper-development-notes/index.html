我是如何做出 TAPD AI 小助手的：从插件方案到实际落地

发表于2026-03-10|更新于2026-03-10|AI Agent
|浏览量:

最近我做了一个自己的小工具：TAPD AI 小助手。
目标很直接：在不改动 TAPD 原有系统的前提下，把“读需求、提建议、改文案、保存回 TAPD”这一整条链路跑通。

这篇文章记录完整开发过程，包括方案选择、架构设计、关键实现和上线步骤。

一、先说结论：为什么我选“浏览器扩展 + 独立工作台”

一开始我考虑过两种路线：

  1. 直接做 TAPD 页面侧边栏增强
  2. 做独立工作台，左边嵌 TAPD 页面，右边放 AI 能力

最后我选了第二种，并且保留了 content script 的页面读写能力，原因有三个：

  • 交互更完整：右侧可以做多 Tab（摘要、质量、历史、技术、对话），不会挤占 TAPD 原页面空间。
  • 改造成本更低：不需要侵入 TAPD 业务代码，通过 iframe + message 就能协作。
  • 可扩展性更好：后续加模型切换、提示词模板、日志面板都更容易。

二、核心架构：三个脚本分工

当前项目里最关键的三个文件是：

  • tapd-ai-helper/app.js：独立工作台主逻辑，负责 UI、状态和与 iframe 通信。
  • tapd-ai-helper/content.js：运行在 TAPD 页面（含 iframe）中，负责提取需求内容、写回编辑器、触发保存。
  • tapd-ai-helper/background.js：统一处理 AI API 请求和 TAPD Open API 请求，避免在页面侧暴露 Key。

实际数据流是这样的：

  1. app.js 加载 TAPD 链接到 iframe
  2. content.js 在 iframe 内读取需求标题/描述并 postMessage 给 app.js
  3. 用户点击“分析/检查/对话”后，app.js 把请求发给 background.js
  4. AI 返回结果后，用户可点击“应用修改”
  5. app.js 再发消息给 content.js 执行 applyDescription，最后可触发 savePage

三、几个关键实现点

1) 让 TAPD 页面能被 iframe 嵌入

TAPD 默认会限制页面被嵌套，所以我在 manifest.json 中启用了 declarativeNetRequest，并配合 rules.json 处理响应头，移除阻碍 iframe 的策略头。

这一步是整个独立工作台能成立的前提。

2) content script 做“双模式”

content.js 我做了两种运行模式：

  • 页面模式：作为普通扩展注入时，可直接在 TAPD 页面使用侧栏能力。
  • iframe 模式：当页面被独立工作台嵌入时，只做“读取 + 回写 + 保存代理”。

这样可以避免逻辑分叉，后续维护一份核心代码即可。

3) 需求读取做“多选择器兜底”

TAPD 不同页面结构和版本下 DOM 会变化，所以读取标题、描述、状态时不能依赖单一选择器。

我在 content.js 里做了多组候选选择器与 URL 解析兜底，比如从链接里补 workspaceId/storyId，这样在页面结构轻微变化时仍能工作。

4) AI 请求放到 background.js

模型请求统一由 background.js 发起，支持多个 provider（OpenAI / Claude / Gemini / Groq）。

这样有两个好处：

  • API Key 不落在页面脚本作用域，安全边界更清晰
  • 不同模型只改一处适配层，前端交互逻辑不用改

5) 历史需求对比做“可选增强”

历史对比功能默认可用（让 AI 基于当前需求做推断），也支持在设置中填 TAPD Open API 凭据后，真实拉取同工作区历史需求做上下文增强。

这让工具既能“开箱即用”，也能“逐步增强”。

四、Prompt 设计：把能力拆成五个场景

我没有用一个超级大 Prompt，而是按场景拆成五组消息模板：

  • 需求摘要：提炼目标、功能点、规划范围
  • 质量检查：查错别字、语序、歧义、缺失
  • 历史对比：识别同类需求可能遗漏项
  • 技术建议：输出实现风险、依赖、边界条件
  • 对话改写：基于上下文连续修改，并支持一键应用

这种拆分的优势是：结果更稳定、交互更可预期，后续也便于单独调优每个能力。

五、我踩过的坑

坑 1：页面看起来加载成功，但数据为空

原因通常是 iframe 加载完成时机早于 TAPD 详情渲染完成。我的处理是：

  • 保留 loading/error 状态
  • 多次尝试读取关键节点
  • 读取不到时给出明确重试提示

坑 2：应用修改后内容没变

核心不是写入失败，而是 TAPD 编辑器没进入可编辑状态。
所以现在会提示用户先进入编辑态，再执行“应用修改”。

坑 3：不同模型输出风格差异大

通过统一 system 指令 + 输出格式要求，把“可直接粘贴回需求文档”的一致性拉齐了很多。

六、上线发布：我实际做了什么

我的发布路径是：

  1. 在 tapd-ai-helper 本地迭代功能
  2. 在 skill-practice/source/_posts 写开发复盘文章
  3. 运行 Hexo 生成静态站点并发布到 gh-pages

这次文章也遵循这个流程，技术内容和项目代码保持同步。

七、下一步计划

接下来我会继续补这几件事：

  • 增加“需求变更 diff 视图”，应用前先预览改动
  • 给历史对比增加可配置时间范围和关键字策略
  • 增加“团队写作规范模板”，让输出更贴合团队习惯

如果你也在 TAPD 里经常写需求文档，这种“左侧业务系统 + 右侧 AI 工作台”的模式非常值得试一次。
它不是替代产品经理，而是把高频、重复、容易漏的工作交给 AI 先跑一遍。
