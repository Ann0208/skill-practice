浏览器渲染原理：从输入 URL 到页面展示的完整链路

发表于2024-02-28|更新于2026-04-29|性能优化
|浏览量:

“从输入 URL 到页面展示经历了什么”是经典面试题，但理解这个过程对性能优化有实际意义。这篇从网络请求到渲染流水线完整梳理一遍。

网络阶段

  1. DNS 解析：域名 → IP 地址。浏览器缓存 → 系统缓存 → 路由器缓存 → DNS 服务器，逐级查找。
  2. TCP 连接：三次握手建立连接。HTTPS 还要多一步 TLS 握手。
  3. 发送 HTTP 请求：请求行 + 请求头 + 请求体。
  4. 接收响应：状态码 + 响应头 + 响应体（HTML 文档）。

解析阶段

浏览器拿到 HTML 后开始解析：

构建 DOM 树：HTML 解析器把标签转换成 DOM 节点，构建树形结构。遇到 <script> 标签会暂停解析（JS 可能修改 DOM），这就是为什么 script 标签要放在 body 底部或加 defer/async。

构建 CSSOM 树：CSS 解析器把样式规则构建成 CSSOM（CSS Object Model）。CSS 不会阻塞 DOM 解析，但会阻塞渲染——浏览器要等 CSSOM 构建完才能渲染，避免出现无样式内容闪烁（FOUC）。

合并为渲染树：DOM + CSSOM → Render Tree。渲染树只包含可见元素（display: none 的不在里面）。

渲染阶段

布局（Layout/Reflow）：计算每个元素的位置和大小。这一步开销大，应该尽量减少触发次数。

绘制（Paint）：把元素画到屏幕上。分层绘制，每一层独立。

合成（Composite）：把各层合成最终画面。transform 和 opacity 的动画只触发合成，不触发布局和绘制，所以性能好。

回流与重绘

  • 回流（Reflow）：元素的几何属性变化（宽高、位置、display），需要重新计算布局。开销大。
  • 重绘（Repaint）：元素的外观变化（颜色、背景、阴影），不影响布局。开销较小。

回流一定触发重绘，重绘不一定触发回流。

减少回流的方法：

  • 批量修改 DOM：用 documentFragment 或先 display: none 再修改
  • 避免逐条修改样式，用 className 一次性切换
  • 读写分离：不要在循环里交替读取布局属性和修改样式
  • 动画用 transform 代替 top/left

关键渲染路径优化

  • CSS 放 head：尽早开始构建 CSSOM
  • JS 放 body 底部：或用 defer（按顺序执行）/ async（下载完立即执行）
  • 减少关键资源数量：内联关键 CSS，延迟非关键 CSS
  • 压缩资源体积：Gzip/Brotli 压缩，图片优化
  • 预加载关键资源：<link rel="preload"> 提前加载字体、关键脚本
